Subsystems

Subsystems are where the more low-level, hardware, tasks are handled. A subsystem represents a group of hardware (drivetrain, arm, etc.) on the robot. Because of this the subsystem handles tasks such as setting motors, controlling pneumatics, toggling LEDs, and setting other types of hardware. Commands will call methods from the subsystems in order to set the hardware based on their input. Below is a basic subsystem skeleton:

    public class ExampleSubsystem extends SubsystemBase {

      public ExampleSubsystem() {}

      @Override

      public void periodic() {}

    }

At the top you can see that the ExampleSubsystem class extends SubsystemBase, similar to commands extending CommandBase. This gives us the periodic() method to override which works the same as other periodic methods in WPILib (it runs periodically, every 20ms by default).

There’s also a constructor for the subsystem. This constructor almost never needs parameters because only one object should be created for each subsystem.

On top of these methods, there will also be methods like setWheelSpeed(), setArmHeight(), and other such methods that will interact with the hardware directly. These will be public methods, called by the commands. In the last lesson, the example command used a doSomething() method from the subsystem class.
