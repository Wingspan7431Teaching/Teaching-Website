Commands

Commands are actions that the robot is instructed to perform. Here, we’ll go over what a file/ class for a command looks like, starting with an example.

    00 //Imports would be up here

    01 public class ExampleActionCommand extends CommandBase {

    02   int exampleInput;

    03

    04   public ExampleActionCommand(int exampleInput) {

    05     //Robot.instance is the robot

    06     //Robot.instance.exampleSubsystem is a subsystem object

    07     addRequirements(Robot.instance.exampleSubsystem);

    08     this.exampleInput = exampleInput;

    09   }

    10   @Override

    11   public void initialize() {

    12     Robot.instance.exampleSubsystem.doSomething(exampleInput)

    13   }

    14   @Override

    15   public void execute() {}

    16   @Override

    17   public void end(boolean interrupted) {}

    18   @Override

    19   public boolean isFinished() {

    20     return false;

    21   }

    22 }

Starting at line 1, notice how the command extends the CommandBase class. This is similar to how our Robot class extends the TimedRobot class which allows us to use methods such as teleopPeriodic(), robotInit(), and other methods. In this case we get to override initialize(), execute(), end(), and isFinished() – which we’ll get to in a second.

Next, on line 4, is the constructor. The constructor initializes (sets) the values for any class variables (line 8), and add subsystem requirements (line 7)

Adding subsystem requirements makes it so that the CommandScheduler never schedules two commands at the same time. If you don’t add subsystem requirements, both commands may try to run at the same time which can cause a lot of problems with the subsystem receiving conflicting instructions.

After the constructor is all of the overridden commands from the CommandBase class. First, there’s initialize(). initialize() runs once when the CommandScheduler first runs the command.

Note: the constructor is run when the command is created (when the robot starts in most cases). The initialize() method will run when the command actually starts. This means that the constructor should be used for doing any tasks that don’t require input from the driver or sensors. On the other hand, the initialize() method should be used for anything that does require such input. For example, if our goal is to move the arm to a specific height, the initial height should be set in the initialize() method because the arm height might change between when the command is created and when it’s run.

Moving on, execute() is run periodically throughout the duration of the command. end(boolean interrupted) is run once when the command finishes. This can be because it completed naturally (isFinished() returned true) or was interrupted by another command. The interrupted variable will say whether the command ended because another command was scheduled for the same subsystem or because the end condition was met.

The end condition is specified in isFinished() which returns true or false depending on if this end condition has been met. For example, the GoToPoint command might return true if the robot is a certain distance away from the desired point.

In summary:

  1. The constructor is run once before the command is even scheduled
  2. Specify which subsystem the command uses in the constructor to avoid scheduling conflicts
  3. initialize() is run once at the start of the command
  4. execute() is run periodically while the command is active
  5. end() is run once when the command finishes
  6. isFinished() specifies the condition that determines when the command should finish
