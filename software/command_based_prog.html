Command-Based Programming (Overview)

Command-Based Programming is a system that defines how we organize our code, manage our robots actions, and perform more complex actions. It’s really useful... But what exactly is it?

Command-Based Programming is a way we split the code into sections. There are two main parts in Command-Based Programming: subsystems and commands. A subsystem is a physical group of parts that performs one functionality (think Drivetrain, Intake, Climber, Shooter, etc.). A command is some action that a subsystem performs (think driving, intaking, climbing, shooting, etc). Most subsystems have multiple different commands. One way to think of it is that subsystems are the hardware and commands are behavior.

In code, we apply command-based programming using classes. A subsystem class controls hardware, a command class tells the subsystem how to set the hardware. Each subsystem gets its own folder, within that folder is a file with the class for the subsystem, and a unique file & class for each command. An example file structure is below:

    src

    └── main

      └── java

        └── frc

          └── robot

            ├── Robot.java

            ├── Main.java

            └── drivetrain

              ├── Drivetrain.java

              ├── DriveWithJoysticks.java

              ├── GoToPointCommand.java

              └── DrivetrainConstants.java

            └── arm

              ├── Arm.java

Example Explantation:

The Robot file at the top is similar to what you’ve been using. This contains all the methods from TimedRobot (robotInit, robotPeriodic, teleopPeriodic, etc.). The main file starts the robot by calling a method (which will be done for you). Then, you get into the Command-Based Programming files.

The first folder has the Drivetrain files. This includes the subsystem, two commands for the subsystem, and a constants file. The constants file holds values that don’t change such as the location of the wheels relative to the robot center, the robot’s max speed, and other such values.

The example arm folder follows a similar structure.

Inside a subsystem file is all the code that directly interacts with the hardware — like spinning motors, turning on LEDs, or performing other low-level actions. Command files, on the other hand, contain the logic that uses input from the user or sensors to call those subsystem methods and perform the necessary actions.

Another key component of Command-Based Programming is the CommandScheduler. The CommandScheduleris a class that handles queuing commands. When you schedule two commands for the same subsystem it either puts it in queue (the second command will run when the first is over) or overrides the first command. This is how we run commands in the Robot.java file.

The commands and subsystems will have methods that run periodically. The CommandScheduler is also in charge of running those. Make sure CommandScheduler.getInstance().run() is in the body of the robotPeriodic() method so the CommandScheduler can run every cycle.

To summarize:

  • The CommandScheduler schedules commands, typically in the Robot.java file.
  • The command processes data that’s input and uses it to call methods from the subsystem.
  • The method(s) in the subsystem physically perform the action.
