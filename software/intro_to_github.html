Intro to GitHub

At a high level, we define GitHub as a version control system that allows collaboration between developers. GitHub stores the code on the cloud so that it can be accessed by multiple people. I like to think of it as the google drive of the coding world: it allows you to work on the same files across multiple devices, and view a version history.

In the last exercise, you:

  • Forked a repository (The term repository refers to a project).
    • Forking a repository allows you to copy a project’s code for your own purposes.
    • It’s like creating a copy of a google doc.
  • Then, you went into GitHub Desktop and cloned this forked repository from your account.
    • Cloning uploads the files that you forked on the cloud to your own computer.

While GitHub can store the code on the cloud, it’s usually better to edit it locally with VSCode. That's why we clone the repository.

Once you have the code on your computer, you can open it in VSCode and make changes to it! However, those edits will only be on your local machine. If you want to share your code with others, you’ll have to send your code back to GitHub. In order to do this you first need to do the following:

  • Create a commit.
  • A commit is a certain set of changes that you’ve made to your code.
    • Since you’ve completed Exercise Zero, you should see a list of file changes on the left side of GitHub Desktop. You can create a title and description for these changes and then commit those changes.
  • Once you’ve created a commit, you can push the commit back to the cloud so it can be viewed from other devices.
  • To view changes from other devices, you’ll need to fetch origin.

A good rule of thumb is to always push your code after making changes. This allows you to view a version history of your changes. It also helps to avoid errors when you begin working with others on the same project. Start by pushing your changes for exercise zero.

The terms and ideas discussed in this lesson may seem foreign for now, and that’s okay. GitHub is something that’s best learned through experience. As you use it more, it will make more sense.

Additionally, many key parts of GitHub were not mentioned in this lesson as they’re not necessary until you’re working on collaborative projects.
