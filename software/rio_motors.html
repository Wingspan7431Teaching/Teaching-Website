The RIO and Motors

In order to understand the programming for a robot, we need to understand the physical robot. For us in programming, the most important part of the robot is the RoboRIO (AKA. the RIO). The RIO is the brain of the robot: it stores and executes all of the code we write. When you deploy your code to the robot, it gets saved to the RIO so that the robot can execute it without needing to be connected to your laptop. You can see a picture of the roboRIO here.

Also note that the RoboRIO is being replaced by the SystemCore for the 2027 season. However, it serves a similar purpose to the RIO. We'll know more by the start of the 2027 season.

On the RIO is a variety of ports that we connect to other pieces of hardware. Additionally, the CAN bus is wired up to the RIO. The CAN bus is a 2-wire system that carries signals from the RIO to the components in order to execute your code. Port numbers are usually from the RIO and CAN/ device IDs are from the CAN bus. The port numbers and IDs are important information that is usually stored on a team document.

One type of hardware that’s wired on the CAN bus is the motor controller. Motor controllers are used to, well, control the motors. When you code a motor, you’re usually coding the motor controller which will then interact with the motor.

The vast majority of subsystems use motors and motor controllers, so understanding how to code them is incredibly important. Moving a robot arm uses motors, the drivetrain uses motors, intake uses motors, and the list goes on.

There are two main types of motors: brushed and brushless. Your code will need this information from you when creating a motor object. In most cases it’s safe to assume that the motors on the competition bot are brushless, but always check to make sure.

Making Motors in Code

There are a variety of different brands of motor controllers, but they all tend to use similar constructors in code. For this lesson I’ll use the SparkFlex motor controller as an example. The constructor for a SparkFlex motor is as follows:

    SparkFlex(int deviceID, MotorType type)

Here, the deviceID is the CAN id for the motor controller and the motorType is whether the motor is brushed or brushless. Here’s an example of creating a SparkFlex motor controller:

    SparkFlex motorController = new SparkFlex(3, MotorType.kBrushless);

This creates a motor controller with a CAN id of 3 that controls a brushless motor. If you want to set the speed of a motor on the motor controller you can use .set(double speed), where speed is a value from -1 to 1. 1 represents the motor at full power and -1 is the motor spinning backward at full power. EX:

    SparkFlex motorController = new SparkFlex(3, MotorType.kBrushless);

    motorController.set(0.5); /*This will set the motor to be spinning at 50% power*/

One last thing to note is that not only are there different types of motors, but there’s also many different brands of motors (with REV and CTRE being the main ones). Different brands have their own libraries for controlling their hardware. Wingspan mostly uses REV products, so the code in these lessons will teach you how to use REVLib. Summary

  1. The RIO is where code is stored on the robot and executed from
  2. The CAN bus is wired to the RIO
  3. CAN ids tell the code which device is which on the robot
  4. Motor controllers control the motors
  5. You can set motors using .set() with a parameter value of 1 for max power forward, or -1 for max power in reverse
